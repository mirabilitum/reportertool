# 工程约束：中文编码与分段处理

## 背景

本项目会长期处理中文 Excel、CSV、JSON、Markdown、HTML 和报告文本。Windows + PowerShell 环境下如果不显式指定编码，容易出现中文乱码、JSON 转义不一致、CLI 输出截断或超时。因此实现前必须固定编码和分段处理规则。

## 编码规则

### 文件读取

- 读取 Markdown、JSON、HTML 时统一使用 UTF-8。
- 读取 CSV 时优先使用 `utf-8-sig`，兼容 Excel 打开的 UTF-8 BOM 文件。
- 读取 Excel 时以 xlsx 内部 XML 的 UTF-8 文本为准，不依赖系统默认编码。
- 不允许依赖 PowerShell、Python 或系统 locale 的默认编码。

### 文件写入

- CSV 输出使用 `utf-8-sig`，便于 Excel 直接打开中文不乱码。
- JSON 输出使用 UTF-8，并设置 `ensure_ascii=False`。
- Markdown/HTML 输出使用 UTF-8。
- 如果文件要被 Excel 直接打开，优先 CSV with BOM；如果文件主要给程序读取，优先 JSON UTF-8 no BOM。

### CLI 和 PowerShell

在 PowerShell 调用外部 CLI 或处理中文输出前，设置：

```powershell
$OutputEncoding = [Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false)
```

避免把大段中文直接作为命令行参数传递。长 prompt、长 JSON、长 Markdown 应通过 stdin、临时文件或分段文件传递。

## JSON 规则

- 所有 JSON 字段必须是合法 JSON，不使用伪 JSON 或自然语言列表。
- Python 写 JSON 时使用 `json.dumps(value, ensure_ascii=False)`。
- CSV 中嵌入 JSON 时，字段内容仍必须能被 `json.loads()` 解析。
- 对数组字段统一使用 JSON 数组，例如：

```json
["630932692943118355", "630932692943101982"]
```

## 分段输入规则

当输入文件较大或内容较多时，不做一次性全量塞入模型或 CLI：

- 文档 review：按 `README/需求/设计/计划/schema` 分组提交。
- 样例数据 review：只提交表头和前 N 行，必要时再提交特定 question_id 或 school 的切片。
- Excel 读取：先读 sheet 列表、首页、题型统计，再按题型抽样读取题号 sheet。
- 大纲处理：按章节分段处理，每次只处理一章或一组图。

## 分段输出规则

输出文件或结果较多时必须分层输出：

- 第一层：README 或索引文件，说明文件用途和阅读顺序。
- 第二层：schema/规则/样例表。
- 第三层：图表预览和报告草稿。
- 第四层：完整日志或长 review 结果。

CLI 输出较长时，写入文件后只在对话中总结关键结论，不直接输出全文。

## 大文件处理规则

- 不对完整 Excel 工作簿做无约束全量加载，优先按 sheet 和行流式读取。
- 对 `base_answer_fact` 这类大表，预览只输出前 N 行和统计摘要。
- 对图表预览，默认只展示被报告规则选中的题号或指标。
- 对 review 或调试，使用切片文件，例如某个 `question_id`、某个学校、某个章节。

## 测试要求

实现计划中必须包含以下测试：

- 中文 CSV 写出后用 `utf-8-sig` 读取，中文字段不乱码。
- JSON 字段用 `json.loads()` 解析成功，且中文保持原文。
- Markdown/HTML 写出后包含正确 `<meta charset="utf-8">` 或 UTF-8 内容。
- 大样例输入时，CLI 可以只生成索引、摘要和抽样预览，不强制一次性打印全部内容。
