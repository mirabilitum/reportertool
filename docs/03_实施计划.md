# 报告生成工具 Implementation Plan

> **For agentic workers:** Steps use checkbox (`- [ ]`) syntax for tracking. Implement task-by-task and keep generated samples small enough for manual review.

**Goal:** 构建一个能读取 Word 问卷题目、读取监测 Excel、生成可连接中间表、执行数据重构、支持交叉分析预览、报告写作输入，并输出 HTML/DOCX 两类最终成品的第一版报告生成工具。

**Architecture:** Word 问卷归一表是题目主字典；Excel 输入适配器负责解析文件标题、读取作答数据，并生成允许字段/维度缺失的 `base_answer_fact`；可选字段/维度映射只补充答案组件和核对信息；统计、图表和文字优先消费 `norm_id` 与重构后指标。HTML 预览只用于口径核对；最终报告由组装层把章节、文字、图表结果同时输出为 HTML 和 DOCX，二者消费同一份报告结构，不能只在 HTML 中实现报告逻辑。

**Tech Stack:** Python 3、`python-docx`、`openpyxl`、标准库 `csv/json/hashlib/pathlib`、静态 HTML/SVG、DOCX、Markdown 文档；可选 `matplotlib/seaborn` 用于复用样例 PNG 图表规范。

## 文件结构

- Keep: `src/questionnaire/`，独立 Word 问卷题目归一模块，负责 DOCX 解析、题目合并、人工标注和归一结果输出。
- Create: `src/reportertool/__init__.py`，包入口。
- Create: `src/reportertool/title_parser.py`，解析 Excel 文件名和首页标题中的项目、角色、学科、学期、表单 ID。
- Create: `src/reportertool/questionnaire_normalize.py`，包装 `src/questionnaire/` 的运行入口，负责 `normalize-questionnaires` 阶段输出归一表和异常清单。
- Create: `src/reportertool/questionnaire_mapping.py`，读取 `normalized_question_mapping_long.csv/json`，提供 `subject + question_no -> norm_id/question_text/canonical_question`。
- Create: `src/reportertool/xlsx_reader.py`，读取 xlsx 的 sheet、行、单元格文本。
- Create: `src/reportertool/input_detect.py`，判断分题 sheet 型和普通宽表型。
- Create: `src/reportertool/metadata.py`，抽取首页、题型统计、字段映射、维度映射；这些信息可缺失。
- Create: `src/reportertool/normalize.py`，转换为 `base_answer_fact`、`question_table`，并允许 `field_id/dimension_id` 为空。
- Create: `src/reportertool/reconstruction.py`，读取数据重构规则并生成 `reconstructed_indicator_fact`。
- Create: `src/reportertool/report_rules.py`，读取报告生成规则，规则优先引用 `norm_id`。
- Create: `src/reportertool/metrics.py`，生成基础信息、作答人数、选项汇总、学校内比例、热力图矩阵、数值指标。
- Create: `src/reportertool/chart_registry.py`，注册统一图表类型和样例别名。
- Create: `src/reportertool/visual_style.py`，沉淀字体、颜色、尺寸、坐标轴、图例、备注和小数位规则。
- Create: `src/reportertool/chart_data.py`，把指标/事实数据适配为渲染器可消费的 `ChartDataset`。
- Create: `src/reportertool/chart_renderer.py`，把 `ChartDataset + ReportRule + VisualStyle` 渲染为 `ChartResult`。
- Create: `src/reportertool/chart_text.py`，基于 `ChartDataset + ReportRule` 生成确定性文字和备注。
- Create: `src/reportertool/preview_html.py`，生成静态 HTML/SVG 口径核对预览，不作为唯一最终输出。
- Create: `src/reportertool/report_assembler.py`，把章节规则、图表结果、确定性文字和质量检查组装为统一 `ReportArtifact`。
- Create: `src/reportertool/report_html.py`，根据 `ReportArtifact` 输出最终 HTML 报告。
- Create: `src/reportertool/docx_writer.py`，根据 `ReportArtifact` 和可选 Word 模板输出最终 DOCX 报告。
- Create: `src/reportertool/review_package.py`，按阶段生成给人工复核的 CSV/HTML/JSON 清单和 `stage_status.json`。
- Create: `src/reportertool/pipeline.py`，编排各阶段运行、阶段暂停、继续和 `run-all`。
- Create: `src/reportertool/cli.py`，命令行入口。
- Local only: `tests/fixtures/`，放最小化 Word/Excel/CSV fixture；该目录不纳入 git，必要时用代码生成 fixture 或另行提供脱敏数据包。
- Create: `tests/test_*.py`，覆盖标题解析、题目映射、标准化、指标计算。

## 阶段边界与人工复核

实现需要支持阶段性运行。人的交互节点是 CLI/pipeline 边界，不是底层算法边界；底层模块仍按题目归一、输入读取、标准化、重构、指标、图表、报告输出职责拆分。

| 阶段命令 | 主要模块 | 人工输入或判断 | 主要产物 |
| --- | --- | --- | --- |
| `check-inputs` | `pipeline.py`, `title_parser.py` | 提供 Word、Excel、可选学校信息表、可选规则和模板；判断缺失资料是否阻断 | `review/inputs_file_list.csv`, `review/input_quality_checks.csv` |
| `normalize-questionnaires` | `questionnaire_normalize.py` 包装外部归一脚本 | 判断同义题合并、错位题、单独抽出题是否正确 | `normalized_question_mapping_long.csv`, `normalized_question_mapping_wide.csv`, `review/questionnaire_anomalies.csv` |
| `inspect-excel` | `title_parser.py`, `input_detect.py`, `metadata.py` | 判断 `role`、`subject`、结构识别和首页冲突 | `review/excel_title_parse.csv`, `review/excel_structure_parse.csv` |
| `normalize-facts` | `xlsx_reader.py`, `normalize.py` | 核对未匹配题、答案组件、字段/维度缺失是否可接受 | `samples/base_answer_fact.csv`, `samples/question_table.csv`, `samples/quality_checks.csv` |
| `reconstruct` | `reconstruction.py` | 确认分值映射、指标公式、选项合并、分母口径 | `samples/reconstructed_indicator_fact.csv`, `review/reconstruction_checks.csv` |
| `build-charts` | `metrics.py`, `chart_data.py`, `chart_renderer.py`, `chart_text.py` | 判断样本量、比例、交叉分析和图表口径是否正确 | `preview/chart_manifest.csv`, `preview/chart_data/*.csv`, `preview/cross_analysis_preview.html` |
| `assemble-report` | `report_rules.py`, `report_assembler.py` | 确认章节、图表顺序、题图文字和确定性段落 | `reports/report_artifact.json`, `review/report_outline.html` |
| `write-report` | `report_html.py`, `docx_writer.py` | 验收 HTML 与 DOCX 内容一致性和 Word 模板效果 | `reports/final_report.html`, `reports/final_report.docx`, `reports/assets/*` |
| `run-all` | `pipeline.py` | 使用已确认配置连续运行全部阶段 | 全部阶段产物 |

每个阶段都应写出 `review/stage_status.json`，至少记录 `stage_name`、`status`、`blocking_issue_count`、`warning_count`、`input_paths`、`output_paths`、`next_human_action`。如果某阶段存在阻断项，后续阶段默认不继续，除非 CLI 显式传入 `--allow-warnings` 或修正配置后重跑。

## 执行能力与 skill 支持判断

完成第一版项目不需要新增 superpowers skill。现有能力已经覆盖主要工作：

- 需求和架构澄清：使用现有 `brainstorming` 和普通对话即可。
- 多步骤实施计划：使用现有 `writing-plans` 足够。
- 方案/计划复核：继续使用现有 `cc-review-workflow`，在本机 Claude CLI 可运行时提交 review；如果 PowerShell 执行策略阻断，需要改用可执行入口或先修复本机 CLI 环境。
- 代码实现、测试、文档更新：由当前 Codex 会话直接完成即可，不需要额外 skill。

暂不建议现在新建专用 skill。原因是报告工具的数据结构、阶段产物和人工复核节点还会在第一轮真实 Excel/Word 上调整，过早抽成 skill 会把未稳定的流程固化。等第一版跑通并至少复用到第二批同结构问卷后，可以考虑新增一个 `reportertool-workflow` skill，把固定命令、复核清单、常见质量检查和交付目录约定沉淀下来。

### 2. 图表规范

- 字体：Python 样例统一 `SimHei` 并关闭 unicode 负号问题；R 样例标题/图题多用 `STKaiti`，坐标和图例常用 `PingFang SC`。实现时抽成 `font_body`、`font_title`、`font_caption`，缺字体时降级到系统中文字体。
- 颜色：不要把地区项目色硬编码到图函数。抽成 palette token：R 样例主色 `#6a99d0/#869EA3/#BFAF6D/#E0AD5B/#D1BABE/#A86D70`，Python 样例分类色 `#1b4551/#4a706f/#629991/#99af86/#ddc781/#dda976/#c87b5e`，热力图蓝/绿/灰红渐变，评价色保留 `达标/不达标/表现好/表现较好/表现待提高` 的显式映射。
- 尺寸：Word/PNG 输出默认宽度 5.5 英寸、300 dpi；已有外部图片可按 6 英寸宽插入，默认高 4.5 英寸。R 样例图高按图表类型或行数动态计算：简单条形约 2.5，饼图 2.5，堆叠条 3 起并随 Y 类别增加，相关矩阵按变量数动态，高度上限 7。Python 样例常用 `(12,6)` 饼图、`(16,6)` 堆叠条、`(12,8)` 折线、`(10,4)` 单折线、热力图 `14 x (1 + cube_h * (行数 + 3))`、学校堆叠条 `18 x 动态高度`、分面图 `(16,5)` 或 `(16,10)`。
- 标题与图题：内部图标题使用 `title`，字号通常 `text_size + 2` 或 14、加粗；R 样例常将 `图题表题` 放到 `labs(caption=...)` 并居中。实现中必须区分 `chart_title` 和 `figure_caption`，Word 输出避免重复写同一标题。
- 坐标轴：比例图统一用 0-100% 语义；Python 用 `PercentFormatter(1)`，R 横向堆叠条常用 0-109 的绘图区、0-100 的刻度。多数报告图隐藏 x/y 轴标题，只保留分类标签和数值标签；长标签必须换行。
- 图例：图例默认底部居中、无边框，标签按宽度换行；饼图可右侧图例；学校数量多时限制每行图例数并给备注留空间。
- 数值标签：比例默认 0 或 1 位小数，均值/分数默认 1 或 2 位小数；0 值和过小值可不标或外置引线；`figures_with_dot` 这类小数保留规则应迁移为 `decimal_policy`。
- 备注：图下注释字号 11，统一以 `备注：` 开头；包含样本量、数据来源、计算口径。备注位置不能写死，渲染后根据图例/坐标标签 bbox 自动下移或预留底部空间。
- 表格：表头背景 `#B7B7B7`、表头居中、正文 1.2 倍行距，表格宽度 100%，长文本列自动适配。

### 3. 文字规范

- 标题层级来自报告规则或 Word 模板：`一级标题`、`heading 2` 到 `heading 5`；正文用 `Body Text`；以 `注：` 开头的段落可用 footer/备注样式；图表标题用 `图表标题`。
- 动态正文只使用确定性模板生成，不让 LLM 参与数值判断。可复用模板包括：
  - 多选/频次题：提取 Top 1/2/3，写作形态为“在 X 方面，选择最多的 N 项为：A（p%）、B（p%）。”
  - 单指标区域/学校比较：写作形态为“评估显示，学生/教师 X 指数/分数为 v%/v。”
  - 堆叠构成：通过 `sum_indices` 指定需要合并解读的分类，写作形态为“在 X 上，p%的学生/教师达到 A 或 B 的水平。”
  - 饼图/全分类：可输出“在 X 上，选择情况为：A（p%）；B（p%）。”
  - 题项均值：输出维度均分、最高题项、最低题项。
  - 样本量：输出“共回收有效问卷 N 份，有效参测比例约为 100%。”这类口径，但“100%”必须来自规则或实际计算，不可默认。
- 图下注释和正文分离：正文解释结论；备注说明样本量、数据来源、计算方法、剔除异常值等。
- LLM 后续只允许做可选润色或报告段落扩写，输入必须包含已计算好的 `writing_metrics_json`，输出不得新增未经计算的数值。

### 4. 数据上下游接口规范

上游输入分三层，图表函数只消费后两层，不能直接依赖原始 Excel：

```text
base_answer_fact:
  必备键: dataset_id, role, subject, user_id, school, region,
          grade, semester, question_no, norm_id, question_text,
          answer_component_id, answer_component_type, answer_component_label,
          field_value, raw_value, source_file, source_sheet, source_row_index

reconstructed_indicator_fact:
  必备键: indicator_id, subject, role, user_id/school, scope fields,
          indicator_value, value_type, source_norm_ids_json, source_fact_ids_json

MetricsSummary:
  必备键: metric_id, metric_type, scope_key_json, role, subject,
          norm_id, indicator_id, category, series, stack,
          count, denominator, percent, mean, sd, n,
          source_norm_ids_json, source_indicator_ids_json, source_fact_ids_json
  说明: 这是 metrics.py 到 chart_data.py 的显式接口；缺少某类数值时列仍保留为空。

report_rule / chart_request:
  必备键: rule_id, chart_type, title, figure_caption, note_template,
          norm_ids_or_indicator_ids, filter_scope_json, compare_scope_json,
          group_by, stack_by, value_column, denominator_policy,
          decimal_policy, text_template_id, writing_metrics_json,
          data_source_text, writing_instruction
  说明: writing_instruction 仅供人工阅读和规则维护，不得作为 LLM 输入；LLM 可选润色只能消费已计算的 writing_metrics_json。
```

`chart_data_builder` 以 `MetricsSummary` 和 `reconstructed_indicator_fact` 为主要输入，输出给渲染器的统一表：

```text
ChartDataset:
  rule_id
  chart_type
  title
  rows: category, series, stack, value, count, denominator, percent, mean, sd, n
  display: decimal_places, percent_scale, sort_order, label_wrap, palette_key
  provenance: source_norm_ids_json, source_indicator_ids_json, source_fact_ids_json
  说明: count 表示当前分类/选项命中数；denominator 表示按 denominator_policy 去重后的分母；n 表示均值、标准差等统计量的有效样本量，比例图可与 denominator 相同。
```

`chart_renderer.render_chart(dataset: ChartDataset, rule: ReportRule, style: VisualStyle) -> ChartResult` 负责把图表数据渲染成 SVG/PNG 或表格化预览；`preview_html.py` 只消费 `ChartResult`，不重新统计、不重新生成图表数据。

渲染器输出：

```text
ChartResult:
  rule_id
  chart_type
  title
  figure_caption
  image_path 或 svg_html
  encoded_png 可选
  plot_data_csv
  text_paragraphs_json
  note
  quality_checks_json
```

`chart_manifest.csv` 字段至少包含：

```text
rule_id, chart_type, title, figure_caption, image_path, plot_data_csv,
text_paragraphs_json, note, quality_status, quality_checks_json
```

最终报告组装层消费 `ChartResult` 和报告规则，生成统一报告结构：

```text
ReportArtifact:
  report_id
  title
  output_formats: html, docx
  chapters:
    chapter_id, chapter_title, sections[]
  sections:
    section_id, section_title, chart_results[], text_paragraphs[], notes[], quality_checks[]
  assets:
    image_path, plot_data_csv, asset_role
  provenance:
    questionnaire_mapping_path, excel_inputs_json, chart_manifest_path, generated_at
```

`report_html.py` 和 `docx_writer.py` 都只能消费 `ReportArtifact`，不得重新统计、重新选择题目或重新生成图表数据。HTML 可以用于成品交付，也可以作为最终检查文件；DOCX 是同等级成品输出，不能依赖人工从 HTML 复制生成。

关键口径：
- `norm_id` 是题目主键；只有需要选项、矩阵行列、上传/文本子项时才使用 `answer_component_id` 或 `answer_component_label`。
- `metrics.py` 是通用聚合口径层；`chart_data.py` 优先消费 `metrics.py` 的聚合结果和 `reconstructed_indicator_fact`，只做图表形状适配，避免重复实现分母、去重和比例口径。
- 学校所在盟市/旗县等细分地区来自 `school_info` 或 scope 配置，`base_answer_fact` 只强制保存 `region`。
- 多选题分母按 `norm_id + scope + user_id/school` 去重，分子按被选中的答案组件计数；一个用户多个选项不能重复进入分母。
- 学校表、教研组表、教师表、学生表必须通过 `role` 和 `denominator_policy` 确定统计单位：学校、教研组、教师、学生。
- Python 样例中的 `alternative` 对应本项目的 `answer_component_label`；`field_value` 保留为原始/数值作答值。
- R 样例中的 `_Score/_Class/_Figure` 对应重构后指标，迁移为 `indicator_id + indicator_value + value_type`，不要在事实表里继续硬编码列后缀。

### 5. 可规则化与可复用内容

- 可直接复用为规则：`chart_type` 注册与别名、palette token、尺寸与 dpi 默认值、百分比坐标规范、图例/备注布局规则、表格样式、样本量/数据来源/计算方法备注模板、TopN/占比/均值/最高最低题项文字模板、`export_plot_data` 的“每张图必须保留绘图数据”思想、HTML/DOCX 共同消费同一份 `ReportArtifact` 的输出约束。
- 可复用但需配置化：地区列表、学科列表、教材选项、达标阈值、分类合并规则、`sum_indices`、小数位规则、异常值剔除规则、报告标题层级。
- 不应进入核心复用层：样例中的本地绝对路径、特定城市/学校修数、具体项目名称、特定题号文案、临时 display/debug 代码。
- 落地时新增 `chart_registry.py`、`visual_style.py`、`chart_data.py`、`chart_renderer.py`、`chart_text.py`、`report_assembler.py`、`report_html.py`、`docx_writer.py`；`preview_html.py` 只负责组装 `ChartResult` 为预览页，不直接做统计，也不承担最终 DOCX 输出职责。

## 自检

- Word 题目归一表是题目主字典。
- Excel 文件标题解析出 `role` 和 `subject`。
- `norm_id` 是跨学科题目主键。
- `field_id`、`field_title`、`dimension_id`、`dimension_name` 允许为空。
- 字段/维度只作为可选答案组件和人工核对信息。
- 报告规则优先引用 `norm_id` 或 `indicator_id`。
- 第一版支持 HTML 与 DOCX 两类最终成品输出；HTML 同时可作为最终检查文件。复杂 Word 版式、PDF 导出和模板精修可后续增强，但不能把 HTML 预览当作唯一交付物。
