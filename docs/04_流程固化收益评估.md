# 流程固化收益评估

## 结论

建议固化，但不要一次性固化全部能力。最值得优先固化的是数据读取、基础事实表、题目表、数据重构规则、重构后指标事实表、复核定位、常见统计口径和预览输出；不建议过早固化最终报告样式和完整文字模板。

当前数据目录包含多个学科、学生、教师、教研组、学校表，且后续会反复出现学校内比例、多题交叉、多选热力图、全体与部分对比。这类任务如果每次手工处理，主要成本不是画图，而是重新理解表结构、确认统计口径、处理特殊题和复核结果。因此固化收益较高。

## 固化收益高的部分

- 数据读取流程：固定读取首页、题型统计、字段映射、维度映射、题号 sheet，可减少重复探索。
- 基础中间表：第一类作答事实表和第二类题目表可以稳定支撑原始题目复核、学校内比例、多题交叉和多选热力图。
- 数据重构规则：题目赋值、权重、量表映射、多题组合应固化为配置，而不是散落在图表代码里。
- 重构后指标事实表：规则稳定后生成可核对中间表，避免后续每次报告生成重复计算复杂公式。
- 题目标识策略：跨表使用 `question_id`，表内展示使用 `dataset_id + question_no`，避免题号重复导致连接错误。
- 复核链路：保留短 `fact_id`，并保留 `source_file + source_sheet + source_row_index + user_id + question_id + field_id` 后，任何图表结果都能回到原始数据。
- 常见统计口径：唯一用户数、学校内比例、多选题选择比例、热力图矩阵、全体与部分对比都适合沉淀为通用函数。
- 预览机制：CSV + HTML 预览能让你先确认口径，再决定是否进入正式图表和报告写作。

## 不宜过早固化的部分

- 第一类表字段范围：学校性质、学校数量、学科、学期、年级等字段应由用户说明或配置决定，不能在代码中固定死。
- 第一类表和第二类表的字段去重：为了人工复核，事实表保留字段标题会更直观；为了数据规范，只保留连接键会更干净。这个需要按使用习惯确认后再定默认。
- 数据重构规则的具体内容：量表分值、矩阵题展开、选项合并、文本填空处理都应先由用户确认，不应由系统强行推断。
- 报告生成规则：章节、图表类型、筛选范围、写作指标应允许用户先选部分章节测试，不应一次性锁死。
- 最终报告视觉样式：后续你会提供可视化处理代码，因此第一版只固化数据接口和预览，不固化最终视觉。
- 报告文字模板：应先积累图表类型和统计指标，再抽象模板，否则容易写成过窄规则。

## 成本与风险

- 初始成本：需要多设计事实表、题目表、数据重构规则、报告生成规则，比直接写单题汇总慢。
- 学习成本：使用者需要理解“底表”和“派生表”的区别。
- 文件体积：事实表保留到用户-题目-字段粒度，数据量会比汇总表大。
- 口径风险：如果第一类表字段范围配置不清，会影响后续交叉分析。

这些成本是可控的。事实表可以按用户说明裁剪字段，派生表可以缓存，数据重构规则可以逐步沉淀，报告生成规则可以按章节测试。

## 判断标准

如果只做一份报告，流程固化的收益一般，手工脚本会更快。

如果要持续处理多个学科、多个对象类型、多个地区或多轮报告，固化收益明显。当前样例目录已经符合这种情况，因此建议把“读取 -> 基础事实表/题目表 -> 数据重构 -> 重构后指标事实表 -> 报告规则 -> 预览 -> 报告输入”作为稳定流程。

## 推荐固化顺序

1. 固化 xlsx 读取和数据结构识别。
2. 固化基础事实表和题目表导出。
3. 固化复核定位字段。
4. 固化学校内比例、多选热力图、跨题用户连接。
5. 固化 CSV/HTML 预览。
6. 固化数据重构规则读取和 `reconstructed_indicator_fact` 输出。
7. 固化报告生成规则读取，支持按章节测试图表类型和写作指标。
8. 等可视化代码和大纲格式稳定后，再固化正式图表和报告文字生成。
