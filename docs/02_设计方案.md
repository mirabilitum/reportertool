# 设计方案

## 推荐方案

第一版采用“Word 题目归一表 + 统一中间表 + 双格式最终报告”方案：先用问卷 DOCX 生成跨学科题目归一对照表，再把不同输入格式转换为可连接的基础作答事实表和题目表。Word 归一表是题目主字典；Excel 字段和维度映射降级为可选的答案组件说明和核对信息。数据层特殊逻辑进入数据重构规则表，并生成重构后指标事实表；报告层要求进入报告生成规则表，最终同时支持 HTML 和 DOCX 成品输出。

这样可以兼容分题 sheet 型 Excel、普通宽表型数据、跨学科同义题对齐、字段缺失题、学校内比例、多题交叉、多选热力图、指标合成和复核定位。

## 备选方案

### 方案 A：直接从原始 Excel 生成图表和文字

优点是开发快，样例表能很快跑通。缺点是逻辑会绑定具体 Excel 结构，普通宽表和后续新格式会不断增加分支，长期维护成本高。

### 方案 B：统一中间表后再生成图表和文字

优点是结构清晰，可测试性强，后续只要保证输入转换正确，图表和文字模块可以复用。缺点是第一版需要多设计几个中间表。

### 方案 C：先生成纯 JSON，再由前端或可视化代码消费

优点是与可视化层解耦。缺点是对非开发人员不够直观，确认统计口径时不如 CSV 和 HTML 方便。

推荐采用方案 B，并同时输出 CSV 和 JSON。CSV 便于人工确认，JSON 便于后续程序消费。

## 数据流

1. 问卷题目归一：独立模块位于 `src/questionnaire/`，通过 `src/reportertool/questionnaire_normalize.py` 和 `reportertool normalize-questionnaires` 暴露阶段入口。从与原始 Excel 相同的资料目录中递归读取 DOCX 问卷，生成 `normalized_question_mapping_long.csv` 和 `normalized_question_mapping_wide.csv`；内部读取为 `questionnaire_mapping_long` / `questionnaire_mapping_wide`，提供 `subject + question_no -> norm_id/question_text/canonical_question` 的连接关系。`norm_id` 是跨学科同义题主键。
2. 输入识别：判断工作簿是分题 sheet 型还是普通宽表型。
3. 文件标题解析：从 Excel 文件名或首页表单标题解析 `project_name`、`form_title`、`subject`、`role`、`semester`、`form_id`。当前数据假设每个 Excel 是单学科单角色文件，优先从文件名第二段表单标题解析 `subject` 和 `role`，首页字段用于校验和补充。
4. 元数据读取：抽取首页、题型统计、字段映射、维度映射。字段和维度映射是可选增强信息，缺失时不阻断主流程。
5. 明细标准化：将每道题数据转换为第一类作答事实表，并通过 `subject + question_no` 补充 `norm_id`、`question_text`、`canonical_question`。
6. 答案组件标准化：如果 Excel 字段或维度映射存在，将选项、矩阵行列、上传状态、填空值等整理为可选答案组件；如果不存在，事实表仍保留 `norm_id + raw_value/field_value`。
7. 字典标准化：将 Word 题目归一表、题型、可选字段、可选维度转换为第二类题目表。
8. 数据重构：读取数据重构规则，对原始事实表进行赋值、公式计算和多题组合。
9. 重构后中间表：生成 `reconstructed_indicator_fact`，作为后续图表、写作和核对的稳定输入。
10. 报告规则读取：读取章节、题号/指标、图表样式、筛选范围、对比范围和写作指标。
11. 指标计算：优先按 `norm_id` 聚合；当规则指定某个选项或矩阵子项时，再使用答案组件字段。
12. 图表结果生成：根据指标生成 `ChartResult`，包括 SVG/PNG 或表格化结果、绘图数据、确定性文字和质量检查。
13. 预览核对：把 `ChartResult` 组装为静态 HTML/CSV 预览，供人工核对口径、题目、样本量和质量检查；预览不是唯一成品报告。
14. 报告组装：基于报告规则、章节结构、`ChartResult` 和确定性文字生成统一 `ReportArtifact`。
15. 成品输出：`report_html.py` 和 `docx_writer.py` 消费同一份 `ReportArtifact`，分别输出最终 HTML 和 DOCX。HTML 可作为最终检查文件，DOCX 是同等级成品交付格式。

## 人工交互与阶段复核节点

人的介入节点应作为流程阶段边界和 CLI 子命令边界，但底层代码模块仍按数据职责拆分。也就是说，外部使用时可以按阶段暂停、复核、补资料；代码内部不能把统计、题目识别、图表渲染等逻辑混在人工流程里。

建议固定以下阶段门：

| 阶段 | 人需要提供或判断 | 自动产物 | 通过条件 |
| --- | --- | --- | --- |
| 输入资料准备 | 提供 Word 问卷、Excel 数据目录、可选学校信息表、可选报告规则和 Word 模板 | 输入清单、文件存在性检查、文件名解析摘要 | 资料齐全；缺失资料被标记为可选或阻断 |
| 问卷题目归一复核 | 判断跨学科同义题是否合并正确，修正错位题、单独抽出的题、题号异常 | `normalized_question_mapping_long/wide`、题目相似候选、异常清单 | 每个需要跨学科比较的题都有稳定 `norm_id` |
| Excel 标题与结构复核 | 判断每个 Excel 的 `role`、`subject`、学期、表单 ID 是否正确，确认分题 sheet 型或宽表型 | 标题解析表、结构识别表、质量检查 | 文件级 `role + subject` 明确；结构识别无阻断错误 |
| 标准化事实表复核 | 核对 `base_answer_fact`、`question_table`、未匹配题、缺失字段/维度、答案组件解释 | 样例事实表、题目表、质量检查表 | 题目能通过 `norm_id` 或降级键定位；字段/维度缺失不会误阻断 |
| 数据重构规则确认 | 确认分值映射、选项合并、指标公式、分母口径、筛选范围 | `data_reconstruction_rules` 校验结果、`reconstructed_indicator_fact` 样例 | 关键指标可回溯到来源题和事实行 |
| 指标与图表预览复核 | 判断样本量、百分比、交叉分析、图表类型和图下注释是否符合口径 | `MetricsSummary`、`chart_manifest.csv`、`preview/cross_analysis_preview.html`、绘图数据 CSV | 图表与表格数据一致；质量检查无阻断项 |
| 报告规则与文字复核 | 确认章节结构、题目/指标引用、图表顺序、确定性文字模板、比较对象 | `report_generation_rules` 校验结果、报告结构预览 | 报告章节和图表清单可生成完整 `ReportArtifact` |
| 最终成品验收 | 判断 HTML 与 DOCX 是否内容一致，是否需要 Word 模板或人工排版微调 | `reports/final_report.html`、`reports/final_report.docx`、资产目录 | HTML/DOCX 章节、图表数量、图题、主要文字一致 |

这些阶段门支持“只跑到某一步再人工看”的实际使用方式，也支持最后一键 `run-all`。每个阶段都应输出可读的 review package，而不是只在终端打印日志。

## 核心中间表

### 第一类表：`base_answer_fact`

统一作答事实表，是内部核心结构。它不只是基础信息，也承担交叉分析的事实底表。字段范围应根据用户提供的说明或配置确定，当前建议保留：

- `source_file`
- `source_sheet`
- `source_row_index`
- `fact_id`
- `dataset_id`
- `project_name`
- `form_title`
- `form_id`
- `role`
- `subject`
- `user_id`
- `school`
- `region`
- `grade`
- `semester`
- `question_no`
- `local_question_key`
- `norm_id`
- `question_type`
- `question_text`
- `canonical_question`
- `answer_component_id`
- `answer_component_type`
- `answer_component_label`
- `field_id`
- `field_name`
- `field_title`
- `dimension_id`
- `dimension_name`
- `field_value`
- `raw_value`
- `status`
- `school_nature`
- `user_scope_tag`

说明：

- `source_file + source_sheet + source_row_index` 用于定位原始 Excel 行。
- `role` 和 `subject` 来自 Excel 文件标题或首页标题。当前数据中每个 Excel 是单学科单角色文件，因此这两个字段在文件级别确定。
- `norm_id` 来自 Word 问卷题目归一表，是跨学科同义题分析和报告规则引用的优先主键。
- `question_no` 用于回溯本学科、本角色原始表内题号；`local_question_key = role + subject + question_no` 用于文件内定位和质量检查。
- `question_text` 保留本学科原始题干，`canonical_question` 保留同义题组代表题干，供图表标题和报告文字默认使用。
- `answer_component_id/type/label` 是可选答案组件，用于描述选项、矩阵行列、标量填空、上传状态或文本答案。
- `field_id`、`field_title`、`dimension_id`、`dimension_name` 均可为空。它们只作为 Excel 字段/维度映射的核对和补充说明，不能作为题目是否可处理的前置条件。
- `field_value` 是标准化后的统计值，`raw_value` 保留原始单元格值。
- 若用户明确不需要某些字段，可以通过配置缩减第一类表，降低文件体积。

### 第二类表：`question_table`

题目表负责解释题组、学科本地题号、题干、可选答案组件、字段和维度。建议结构：

- `entity_type`
- `dataset_id`
- `role`
- `subject`
- `question_no`
- `local_question_key`
- `norm_id`
- `question_type`
- `question_text`
- `canonical_question`
- `display_title`
- `answer_component_id`
- `answer_component_type`
- `answer_component_label`
- `field_id`
- `field_name`
- `field_title`
- `dimension_id`
- `dimension_name`
- `dimension_group_name`
- `dimension_title`
- `join_key`
- `option_order`

题目本体优先来自 Word 问卷题目归一表。字段和维度映射保留，但职责降级为解释选项、矩阵行列、学校/区域/用户等 scope 字段，以及辅助人工核对。字段或维度缺失时，题目表仍应保留题目层记录。

题目标识规则：

- `norm_id` 是跨学科同义题主键，来自 Word 问卷题目归一表。
- `local_question_key = role + subject + question_no` 是本地题目定位键。
- `question_no` 只在单个表格、单个学科角色文件内部有效，不能跨学科直接连接。
- `question_text` 是本学科原始题干；`canonical_question` 是同义题组代表题干，默认用于图表和报告文字。
- 如果某题无法在题目归一表中找到，应输出质量检查记录，并临时使用 `local_question_key` 作为降级连接键，不能静默丢弃或误并入其他题。

### 第三类 A：`data_reconstruction_rules`

数据重构规则表记录“如何把原始事实重构为稳定指标”。它作用在数据层，不关心报告章节。建议结构：

- `rule_id`
- `rule_layer`
- `source_norm_ids_json`
- `target_indicator_id`
- `target_indicator_name`
- `transform_type`
- `group_by_json`
- `value_mapping_json`
- `aggregation_method`
- `normalization_method`
- `formula_json`
- `output_grain`
- `output_value_type`
- `description`
- `user_confirm_required`

典型规则包括：

- 量表选项到分值的映射。
- 某道题从默认 0/1 改为 0.2 或其他权重。
- 多道题合成一个指标。
- 矩阵题展开方式。
- 填空题数值化方式。
- 跨题分析的连接键。
- 选项合并或重命名。

### 第三类 B：`reconstructed_indicator_fact`

数据重构规则稳定后，必须生成重构后指标事实表。它是后续报告生成和人工核对的桥梁。建议结构：

- `recon_id`
- `source_rule_id`
- `indicator_id`
- `indicator_name`
- `output_grain`
- `user_id`
- `school`
- `role`
- `subject`
- `grade`
- `semester`
- `indicator_value`
- `indicator_value_label`
- `source_norm_ids_json`
- `source_fact_ids_json`
- `source_row_indexes_json`
- `audit_note`

这张表的作用是把“规则计算结果”固定下来。后续如果图表或文字有争议，可以先核对这张表，再回溯到原始事实表。

### 第四类表：`report_generation_rules`

报告生成规则表记录“报告怎么生成”。它作用在报告层，不改变原始数据。用户可以先选某些章节测试图表类型，并修改写作指标。建议结构：

- `report_rule_id`
- `rule_layer`
- `chapter_id`
- `chapter_title`
- `section_id`
- `section_title`
- `data_source`
- `norm_ids_or_indicator_ids`
- `chart_type`
- `test_enabled`
- `filter_scope_json`
- `compare_scope_type`
- `compare_scope_json`
- `writing_metrics_json`
- `writing_instruction`
- `user_editable`

报告规则应优先引用 `norm_id` 或重构后的 `indicator_id`。只有当图表需要指定某个选项、矩阵行或答案组件时，才引用 `answer_component_id`、`field_id` 或 `dimension_id`。

## 统计规则

- 作答人数：默认按 `norm_id + school + user_id` 去重计数；如果是单文件内部统计，可退回 `local_question_key + school + user_id`。`question_no` 只用于表内展示和回溯。
- 单选题/多选题：默认按 `answer_component_label` 或 `field_title` 汇总；字段缺失时按 `raw_value/field_value` 做频次统计。
- 量表题：默认按选项或标量值汇总；如果后续需要均值，应通过选项分值映射生成。
- 填空题：优先识别数值字段并计算均值、最大值、最小值、极差、中位数、众数、分布；文本字段先做频次和空值率。
- 上传题：默认统计是否上传、上传数量或文件类型；没有字段映射时仍可按 `norm_id` 统计上传状态。
- 部分数据报告：同时计算全体统计值和筛选后统计值，并输出差异。
- 学校内单题比例：按 `school + norm_id + answer_component_id` 求和；如果没有答案组件，则按 `school + norm_id + field_value/raw_value` 汇总。分母为该 `school + norm_id` 下唯一 `user_id` 数。
- 多选题热力图：通常生成 `school x option` 或 `dimension x option` 矩阵，值可选选择人数、选择比例或均值。
- 多题交叉：以 `user_id` 为连接键，把多题结果转为用户级宽表，再做交叉表、比例、热力图或分组均值。
- 重构指标：优先从 `reconstructed_indicator_fact` 读取，避免每次报告生成时重复计算复杂公式。

## 图表预览与最终输出设计

第一版预览采用静态 HTML，不依赖服务，定位是 QA 和口径核对：

- 条形图：适合选项汇总、学校作答数。
- 简易分布图：适合数值填空题。
- 表格预览：适合基础信息和题目字典。

正式报告输出与预览分层：

- `preview_html.py` 只消费 `ChartResult`，生成 `preview/cross_analysis_preview.html`、`chart_manifest.csv` 和图表数据链接，用于检查。
- `report_assembler.py` 把章节规则、图表结果、确定性文字、备注和质量检查合并成 `ReportArtifact`。
- `report_html.py` 输出 `reports/final_report.html`，可作为最终成品和最后检查文件。
- `docx_writer.py` 输出 `reports/final_report.docx`，支持可选 Word 模板；第一版至少写入标题层级、正文、图题、备注、PNG 图表和表格化结果。

后续正式图表可切换为你提供的可视化代码，只要继续消费同一套中间表并输出同一类 `ChartResult` 即可。

## 流程固化收益评估

这套流程值得固化，但应分阶段固化。

高收益部分：

- Word 问卷题目归一值得固化。它把跨学科同义题、题号差异和题干统一为可复用的 `norm_id`，能减少后续报告规则中重复维护每个学科题号的成本。
- 数据读取与标准化值得固化。不同报告都会重复读取首页、题型统计、字段映射、题号 sheet，并需要复核定位。
- 基础事实表、题目表、数据重构规则和重构后指标事实表值得固化。它们把题目解释、原始作答、数据层规则和报告层规则分开，能稳定支撑柱状图、热力图、交叉分析和文字生成。
- 常见统计口径值得固化。例如唯一用户作答数、学校内比例、多选题选择比例、全体与部分对比。
- 预览输出值得固化。HTML/CSV 预览能让非开发人员快速确认口径，减少后期返工。
- 最终报告输出链路值得固化。HTML 和 DOCX 应消费同一份 `ReportArtifact`，避免出现“HTML 里有逻辑、Word 另做一套”的分叉。

不宜过早固化的部分：

- 第一类表最终字段范围不宜固定死，应由用户说明、项目配置或学校信息表决定。
- 数据重构规则不宜写死在代码里，应放入 `data_reconstruction_rules`，稳定后生成 `reconstructed_indicator_fact`。
- 报告生成规则不应与数据重构混在一起，应放入 `report_generation_rules`，供用户按章节测试和修改。
- 最终报告复杂视觉样式不宜在第一版完全固化，因为后续会接入你提供的可视化代码；但 HTML/DOCX 双格式输出能力需要在第一版固定为架构边界。
- 报告文字模板不宜完全固化，应先沉淀图表类型和统计指标，再抽象模板。

收益判断：

- 如果只做一次报告，完整固化收益有限，手工处理会更快。
- 如果会反复处理多个学科、多个角色、多个地区，固化收益很高，因为最耗时的是数据结构理解、口径复核和交叉分析复用。
- 当前数据目录已经包含大量学科、教师、学生、教研组、学校表，明显属于可复用场景，因此建议固化读取、标准化、中间表和预览流程。

## 报告文字设计

文字不直接“看图”生成，而是与图表共用同一份统计数据。每个图表节点输出：

- 图表标题，默认来自 `canonical_question`。
- 统计对象。
- 样本量。
- 主要结论。
- 全体与部分对比。
- 可选指标：均值、极差、众数、分布集中度、最高/最低项。

这样能保证文字和图表一致，并降低图像识别带来的不确定性。

## 边界控制

第一版需要直接生成最终 HTML 和 DOCX，但不追求复杂排版系统。重点是把数据读准、口径定准、预览核对跑通，并保证 HTML/DOCX 来自同一份报告结构。PDF 导出、复杂 Word 模板、跨页精细排版和人工审美精修可以后续增强。
